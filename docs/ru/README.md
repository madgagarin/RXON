**RXON** (Reverse Axon) — это легковесный протокол межсервисного взаимодействия с реверсивным подключением, разработанный для архитектуры **[HLN (Hierarchical Logic Network)](https://github.com/avtomatika-ai/hln)**.

Он служит "нервной системой" для распределенных мультиагентных систем, соединяя автономные узлы (Холоны) в единую иерархическую сеть.

## 🧬 Биологическая метафора

Название **RXON** происходит от биологического термина *Axon* (отросток нейрона). В классических сетях команды обычно идут "сверху вниз" (Push-модель). В RXON инициатива подключения всегда исходит от подчиненного узла (Воркера/Shell) к управляющему (Оркестратору/Ghost). Это "обратный нерв", который прорастает снизу вверх, создавая канал, по которому затем спускаются приказы.

## ✨ Ключевые особенности

-   **Подключаемые транспорты**: Полная абстракция от сетевого слоя. Один и тот же код может работать через HTTP, WebSocket, gRPC или Tor.
-   **Универсальная система событий**: Единый механизм для передачи прогресса, алертов и любых кастомных сигналов в реальном времени.
-   **Цепочка идентичности Zero Trust**: Встроенная поддержка безопасного "всплытия" событий с полями `origin_worker_id` и `bubbling_chain` для предотвращения подмены отправителя в многослойных холархиях.
-   **Детальная телеметрия ресурсов**: Хартбиты включают гранулярные метрики потребления CPU, RAM и специализированных устройств (GPU, TPU, NPU), включая температуру и уровни памяти.
-   **Приоритеты и дедлайны задач**: Встроенная поддержка `priority` и `deadline` (timestamp) выполнения, что позволяет реализовать умное локальное планирование и авто-отмену устаревших задач.
-   **Ядро без зависимостей**: Сердце протокола не имеет внешних зависимостей (стандартные транспорты используют `aiohttp` и `orjson`).
-   **Строгие контракты навыков**: Все сообщения (задачи, результаты, хартбиты) описывают свои структуры данных через JSON-схемы, что позволяет выполнять автоматическую валидацию и умную диспетчеризацию.
-   **Поддержка Blob-хранилищ**: Встроенная поддержка оффлоадинга тяжелых данных через S3-совместимые хранилища (`rxon.blob`).

## 🏗 Архитектура

Протокол разделен на два основных интерфейса:

1.  **Transport (сторона Воркера)**: Интерфейс для инициации соединений, получения задач, отправки событий и результатов.
2.  **Listener (сторона Оркестратора)**: Интерфейс для приема входящих соединений и маршрутизации сообщений в движок оркестрации.

### Высокосигнальная телеметрия (Heartbeats)

Хартбиты RXON предоставляют Оркестратору детальную картину состояния Холона:
-   `ResourcesUsage`: Потребление CPU и RAM в реальном времени.
-   `DeviceUsage`: Нагрузка на устройства, память и температура для акселераторов.
-   `hot_cache`: Список артефактов/моделей, загруженных в память в данный момент.

### Контракты навыков

Каждый навык, объявленный в RXON, теперь может включать:
-   `input_schema`: JSON-схема для параметров.
-   `output_schema`: JSON-схема для результатов.
-   `events_schema`: Маппинг имен событий на их JSON-схемы.
-   `output_statuses`: Список допустимых логических исходов (например, `success`, `retry_later`).

## 🛡️ Обработка ошибок

RXON использует иерархию исключений на базе `RxonError`. Также определены стандартные коды ошибок для результатов задач:
-   `CONTRACT_VIOLATION_ERROR`: Выходные данные воркера не соответствуют заявленной схеме.
-   `DEPENDENCY_MISSING_ERROR`: Отсутствует необходимая зависимость (артефакт, файл, сервис).
-   `RESOURCE_EXHAUSTED_ERROR`: Исчерпаны физические ресурсы (RAM, VRAM, CPU).
-   `LIMIT_EXCEEDED_ERROR`: Превышены логические лимиты (квоты, окна контекста, рейты).
-   `TIMEOUT_ERROR`: Воркер не успел завершить задачу вовремя (или истек дедлайн).
-   `LATE_RESULT`: (Ответ) Оркестратор отклонил результат, так как дедлайн истек.

## 🧪 Тестирование

Библиотека включает `MockTransport` для тестирования Воркеров в изоляции без запуска реального Оркестратора.

```python
from rxon.testing import MockTransport

# Используйте стандартную фабрику со схемой mock://
transport = create_transport("mock://", "test-worker", "token")
await transport.connect()

# Внедряйте задачи напрямую
transport.push_task(my_task_payload)
```
